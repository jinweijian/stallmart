# Code Review 检查清单

本文用于开发者自查和 Reviewer 审查。发现问题时按严重程度标记。

## 严重程度

| 等级 | 含义 | 处理 |
| --- | --- | --- |
| Critical | 安全漏洞、数据损坏、无法启动、核心链路崩溃 | 必须修复后才能合并 |
| High | 主要功能不可用、接口契约破坏、权限缺失 | 必须修复后才能合并 |
| Medium | 用户可见问题、重要边界缺失、明显维护风险 | 建议本轮修复 |
| Low | 清洁度、局部重复、轻微体验问题 | 可排期 |

## 通用检查

- 是否符合对应模块 README。
- 是否引入本机绝对路径。
- 是否提交生成产物、依赖目录、本地私有配置。
- 是否泄露密钥、token、密码。
- 是否同步更新文档。
- 是否有必要的错误处理。
- 是否有可验证的测试或手动验证说明。

## 小程序检查

- 页面是否注册到 `app.config.ts`。
- 页面路径、目录、文件名是否一致。
- API 是否走 `utils/request.ts`。
- endpoint 是否统一在 `app-config/index.ts`。
- Store 是否只承担跨页面状态。
- loading、empty、error、disabled 状态是否齐全。
- 底部固定栏是否考虑安全区。
- 文案是否简洁，不出现调试文本。
- 图片路径是否存在，例如默认头像应指向 `static/`。

## 后端检查

- Controller 是否只做 HTTP 层工作。
- Service 是否承担业务规则。
- Mapper 是否只做数据访问。
- Params/DTO/Entity 是否边界清晰。
- 是否使用统一响应 `Result<T>`。
- 成功码和错误码是否与前端一致。
- 需要认证的接口是否真的受保护。
- 是否存在越权访问风险。
- 是否存在 SQL 拼接。
- 是否记录了必要日志且无敏感信息。

## API 一致性检查

- 后端路由存在。
- 小程序 endpoint 存在且命名一致。
- 请求体字段一致。
- 响应字段一致。
- 成功码一致。
- 401/403/404/500 的前端处理明确。
- Swagger/OpenAPI 与代码一致。

## 数据库检查

- 初始化脚本是否权威且唯一。
- 表字段与 Entity 一致。
- 逻辑删除字段是否一致。
- 索引是否覆盖主要查询。
- JSON 字段结构是否有文档。

## 必须打回的情况

- 成功码或 endpoint 明显不一致。
- 需要登录的接口被公开放行。
- 提交 `.env` 或真实密钥。
- 页面直接硬编码 API base URL。
- 新增模块但没有更新 docs。
- Docker 编排引用不存在目录且未说明。
