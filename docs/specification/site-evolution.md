# 个人官网长期演进规范

`site/` 是大白的个人技术官网，不是一次性作品。每次更新都应该把真实项目进展沉淀成更清楚的技术表达，而不是堆叠夸张成果。

## 核心定位

- 实战全栈与架构深度并重。
- 用工程闭环表达能力：需求建模、前端体验、后端服务、数据与部署、测试验证。
- 当前项目仍处于早期阶段，内容重点是技术判断、设计过程、验证方式和复盘收获。
- 不编造生产规模、性能数字、业务成果或外部评价。

## 更新触发词

当用户说以下任意表达时，默认按本文更新 `site/`：

- 更新个人站
- 更新 site
- 更新个人官网
- 把最近进展加到个人站
- 优化个人站内容
- 加深个人站技术表达

项目级 Codex skill 位于：

```text
.codex/skills/stallmart-site-update/
```

该 skill 应随仓库提交，不放在全局技能目录中。

## 更新来源

更新前先从真实项目状态找素材：

- 最近 git diff / commit。
- `docs/specification/` 中新增或变化的设计、规范、重构计划。
- `docs/guide/`、`docs/deploy/` 中配置、部署、测试变化。
- `server/`、`app/`、`web/`、`site/` 中已经落地并通过验证的实现。
- 本轮对话里用户明确补充的技术判断或经历。

## 内容分流

| 素材类型 | 更新位置 | 写法 |
| --- | --- | --- |
| 新的架构边界、模块治理、接口治理、数据演进、部署约束 | 匿名架构治理案例 | 问题背景、约束、决策、实现、验证、复盘 |
| 新的学习、源码阅读、调试、质量实践、配置经验 | 技术笔记 | 核心问题、技术拆解、实践清单、常见坑、延伸方向 |
| 多次出现并逐渐稳定的工程能力 | 能力地图 | signal、summary、coreStack、expandedStack、evidence |
| 页面表达不再匹配当前水平 | 首页 hero 或排序 | 更准确地描述当前阶段和技术主张 |

优先加深已有条目。只有当新素材有独立主题时，才新增案例或笔记。

## 早期项目表达规则

早期项目没有大量生产细节时，允许写：

- 如何拆边界。
- 如何设计接口契约。
- 如何确定配置和部署事实来源。
- 如何用测试约束内容、路由、构建产物和质量边界。
- 如何从源码阅读或调试里提炼工程判断。

不要写：

- 没有验证过的性能收益。
- 没有事实支撑的规模和稳定性结论。
- 商业、交易、客户、营收、转化、套餐、SaaS 等营销叙事。
- 真实密钥、密码、token、微信密钥、私有账号或其他敏感信息。

## 默认执行流程

1. 阅读 `AGENTS.md`、`site/README.md`、`docs/specification/site.md`、本文。
2. 检查 `site/src/profile.ts` 和 `site/src/content.ts` 的当前结构。
3. 根据最近项目事实选择内容分流。
4. 若新增 slug、数量、路由或内容约束，先更新 `site/src/profile.test.ts`。
5. 修改内容和必要页面结构。
6. 需要时同步 `docs/specification/site.md` 或 `site/README.md`。
7. 运行：

```bash
npm --prefix site test -- --run
npm --prefix site run build
```

8. 构建后确认 `site/dist/` 和 `site/dist/favicon.ico` 仍存在。

## 维护原则

- 真实比丰满重要。
- 深度来自决策过程，不来自形容词。
- 广度来自工程闭环，不来自技术名词堆叠。
- 每次更新都应该让下次更容易继续，而不是制造一次性文案。
